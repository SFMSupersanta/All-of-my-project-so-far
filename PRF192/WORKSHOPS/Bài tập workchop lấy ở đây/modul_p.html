The C language provides a mechanism for identifying the design units of a program.  A design unit is typically a set of logic constructs or statements that are related to one another.  A trivial example of a design unit is the simple code block that begins with main in our original program: 


 /* One module program      */
 /* mainModule.c            */
 /* BTP100                  */
 /* Jan 31 2005             */

 main() {
     printf("This is BTP100\n"); 
 }

The name of the design unit followed by a pair of parentheses - in this case, main() - precedes the code block.  The name printf refers to a separate design unit that main() itself accesses. 


Structured Design

In designing a program, we subdivide the problem conceptually into a set of design units.  We call these design units modules.  In subdividing the problem, we reduce the number of factors with which to deal simultaneously. 

We can subdivide a problem in different ways.  Ideally, we do so in such a way that each module focuses on some narrower aspect of the problem structure.  A set of modules each of which focuses on a narrower aspect of the problem structure reduces the degree of complexity of the solution.

We select each module so that it

  • is easy to upgrade,
  • contains a readable amount of code, and
  • can be used as part of the solution to some other problem.

In designing a module, we

  • select a descriptive identifier for the module,
  • identify the tasks to be performed within the module.

In a structured design,

  • each module has one entry point and one exit point,
  • each module is highly cohesive, and
  • each module exhibits low coupling.

Cohesion

Cohesion is a measure of the focus within a module.  We say that a module is highly cohesive if it performs a single task.  We say that a module is low in cohesion if it performs a collection of unrelated tasks. 

In designing a cohesive module, we ask whether a certain task belongs: the reason to include it is that it is related to the other tasks in some particular manner.  A reason to exclude it is that it is unrelated to the other tasks. 

Glenford Myers (1978) developed the following degrees of cohesion

  • low cohesion - generally unacceptable
    • "coincidental" - unrelated tasks
    • "logical" - related tasks of which only one is performed - the module identifier suggests a choice
    • "temporal" - multiple logically unrelated tasks that are only temporally related
  • high cohesion - generally acceptable
    • "communicational" - the tasks share the same data - all tasks are carried out each time
    • "sequential" - multiple tasks in a sequentially dependent relationship - output of one task serves as input to another task - the module identifier suggests an assembly line
    • "functional" - performs a single specific task - the module identifier suggests a precise verb phrase

For example, the degree of cohesion for a module that

  • calculates the PST and GST on a sale,
  • checks if a barcode is valid, and
  • raises an integer to the power of another integer
is coincidental.  The module has low cohesion.  For an improved design, we identify each task as a separate module.

Coupling

Coupling is a measure of the degree of interrelatedness of a module to its referring module(s).  We call a module low in coupling if it performs its tasks on its own.  We call a module highly coupled if it shares that performance with some other module including the referring module. 

In designing for low coupling, we ask what kind of data to avoid passing to the module. The data classifications include

  • "data" - used by the module but not to control its execution
  • "control" - controls the execution of the module
  • "external" - part of an environment external to the module that controls its execution
  • "common" - part of a global set of data
  • "content" - accesses the internals of another module

For example, a control flag that tells a module whether to accept

  • integer input from the user, or
  • floating-point input from the user
controls the execution of the module from outside the module.  A module that accepts such a control flag is highly coupled.  For an improved design, we create two separate modules - one that accepts integer input and one that accepts floating-point input, neither of which receives any control flag.

Functions

In C, we represent a module by a function.  A function may receive data and may return a value.  We refer to the description of the internal logic of a function as the function's definition. 

Function Definition

A function definition consists of a header and a body.  The body is the code block that contains the module instructions and generates the return value.  The header, which precedes the body, includes the module name and the list of parameters that receive the module data enclosed within parentheses.  The parameters are variables with their own memory locations.  The syntax of a function definition is 

 returnDataType functionIdentifier ( dataType parameterIdentifier,
                        dataType parameterIdentifer, ... ) {

     /* declarations here */

     /* logical constructs here */

     return /* return expression here */ ;
 }
This is the prototype form of a function definition.  returnDataType is the data type of the value returned by the function.  functionIdentifier is the name of the module.  The parameterIdentifiers between the parentheses are the names of the parameters.  The dataTypes are the data types of the parameters. 

For example, 


 /* power raises base to the power of exponent */ 
 int power ( int base, int exponent ) { 
     int result, i;

     result = 1;
     for (i = 1; i <= exponent; i++)
         result = result * base;

     return result;
 }

power is the name of the function, which returns an int value.  base and exponent are the function parameters, both of int data type. 

void Functions

To identify a function that does not return any value, we specify void for the return data type and exclude any expression from the return statement:  


 return;

Alternatively, we can omit the return statement altogether.

A function that does not return a value is called a subroutine or procedure in other languages.

main

The main function of any program serves a unique purpose.  The main() function is the function to which the operating system transfers control at the start of execution. 

main returns a value to the operating system upon completing execution.  The return value is of int data type.  For completeness, we write 


 int main() {

     /* declarations here */

     /* logical constructs here */ 

     return 0;
 }

The operating system typically accepts a value of 0 as an indicator of success and may use this value to control subsequent execution of other programs.

C compilers assume an int where we don't provide a return data type. 

Calls

We use function calls to transfer execution control to functions.  The syntax of a function call is 

 functionIdentifier ( argument, argument, ... )
functionIdentifer is the name of the function called.  Each argument within the parantheses refers to a data value passed to the function.  An argument may be any expression (with certain exceptions).  The number of arguments in a function call should match the number of parameters in the function definition. 

Once a function finishes executing its own instructions, it returns control to the point immediately following the initiating call. 

Pass By Value

A C function receives parameter data by value; that is, the function receives copies of the data supplied by the arguments in the function call.  More specifically, the compiler allocates space for each parameter and initializes each parameter to the value of the corresponding argument in the function call.

For example, the code 


 /* power raises base to the power of exponent */ 
 int power ( int base, int exponent ) { 
     int result, i;

     result = 1;
     for (i = 1; i <= exponent; i++)
         result = result * base;

     return result;
 }

 int main () {
     int base, exp, answer;

     base = 2;
     exp = 4;
     answer = power ( base, exp ); 
     printf("%d^%d is %d\n", base, exp, answer); 

     return 0;
 }

The pass by value mechanism localizes consequences:  a function may change the value of one of its parameter many times, but the value of the corresponding argument remains unchanged.  In other words, the function itself cannot change the value of an argument in any call to that function.  In this sense, variables used as arguments to the function, which are outside the function proper, are relatively secure. 

Coercion

.  If there is a mismatch between the data type of an argument and the data type of the corresponding parameter, the compiler, wherever possible, coerces the value of the argument into the data type of the parameter.  The following call 


 int answer;
 answer = power ( 2.5, 4 );

is equivalent to 

 int answer, temp;
 temp = 2.5;
 answer = power ( temp, 4 );

The compiler adds the code necessary to coerce 2.5 into an int (2) before initializing the parameter base to the result of this coercion.  So, the value returned by power and assigned to answer is still 16. 

No Parameters

If a function receives no data, we can instruct the compiler to check that each call to that function contains no arguments.  We do so, by inserting the keyword void between the parentheses in the function header.  


 void foo ( void );

If a function receives no data and we omit the keyword void in its function header, 

 void foo (  );

the compiler will NOT check that each call to the function has no arguments.  The compiler will ignore any argument that appears in any call to the function.

Prototypes

Function prototypes describe the form of a function without specifying the implementation details.  A compiler needs certain minimum information to process a function call.  To validate the number of arguments, the compiler needs to know the number of parameters.  To coerce argument values, the compiler needs to know the data types of the parameters.  To determine the data type of the call expression, the compiler needs to know the return data type of the function.

A function prototype provides these three pieces of information.  The syntax of a prototype declaration is 

 returnDataType functionIdentifier ( dataType [parameterIdentifier],
                        dataType [parameterIdentifier], ... );
A function prototype ends in a semi-colon and has no function body.  The parameter identifiers are optional. 

The prototype for our power function is 


 int power ( int, int );

We place function prototypes near the head of our source file and before any function definitions.  If we declare a prototype for each function, we can arrange our function definitions in any order that we prefer.  For instance, we can define our functions after our main function. 


#include

We can collect function prototypes into file (along with any #define directives) and refer to the file in our program.  Such a file is called a header file.  We use the #include directive to instruct the compiler to insert a copy of the header file into our source code. 

The syntax of the #include directive is 


 #include "filename"  /* filename is in current directory */ 

The compiler searches for the named header file in the current directory. 

The alternative syntax of the #include directive is 


 #include <filename>  /* filename is in system directories */ 

With this version, the compiler searches for the named header file in the system directories. 

For example, we include the prototypes for the printf and scanf library functions by adding 


 #include <stdio.h>

 int main ( ) {

     printf("This is BTP100\n"); 

     return 0;
 }


Style

For style, we

  • declare a prototype for each function definition
  • specify the return data type in each function definition
  • specify void for a function with no parameters
  • avoid calling the main function recursively
  • include parameter identifiers in the prototype declarations
  • use generic comments and variables names so that we can use the function in a variety of applications without modifying its code
We apply the rules of structured programming to function design.  We avoid multiple returns and redesign multiple return logic to yield a single return (use flags, if necessary).

We document our function so that it appears as one level of abstraction above our calling function.  We precede each function header with a comment that describes what the function does (not how), what the function needs (in terms of its parameters) and what the function returns (if anything).  We state clearly any assumption or constraint regarding the function parameters in the header comment, so that we need not check the code later if we port it.

For example, 


 /* power returns the value of base raised to 
 *  the power of exponent (base^exponent)
 *  power assumes that the exponent is
 *  non-negative
 */
 int power ( int base, int exponent ) {
     int result, i;

     result = 1;
     for (i = 1; i <= exponent; i++)
         result = result * base;

     return result;
 }
