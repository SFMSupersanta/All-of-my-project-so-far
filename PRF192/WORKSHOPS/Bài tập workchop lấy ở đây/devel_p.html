First and foremost, software development is about getting it right.  Software development is not about getting something that sort of works out on time, but about getting it right, even if it takes more time to do so.  This is a discipline where perfection is crucial. 


Life Cycle

The life cycle of work of software extends from concept inception through testing and maintenance to retirement.

About half of the maintanence time is often spent understanding the source code and the documentation.

Quality

For a program to have longevity, it helps to have a quality design.

What makes a quality design:

  • External Features:
    • Usability
    • Portability
    • Correctness
    • Reliability
  • Internal Features:
    • Maintainability
      • Understandability
      • Testability
      • Modifiability
    • Reusability
    • Efficiency
    • Security
  • Some tools we use to achieve these features:
    • structured programming
    • object-oriented programming
    • standards compliance
    • documentation


Waterfall Model

Check out this Computerworld article about software development.

The waterfall model is the traditional model of the software life cycle.  The waterfall model is a linear model that progresses from ideas to maintenance, step by step. 

The successive steps in the waterfall model are:

  • Concept Planning
  • Requirements Analysis and Definition
  • Design
  • Coding
  • Testing
  • Deployment
  • Maintenance
  • Retirement

Implementation Examples:

  • U.S. Department of Justice
  • Earth System Modeling Framework (NASA)

Deficiencies

Notable deficiencies of the waterfall model include:

  • the requirements are seldom fully known at the outset and grow throughout the software life cycle
  • the user does not see the result until it has been completed
  • the user provides little or no feedback in the development process


Open Source Model

The Open Source model is an incremental, non-hierarchical model.  Its focus is user-driven rather than concept driven.  The rise of the Internet enabled this approach.  Advantages over the waterfall model include:

  • reliability
  • robust functionality
  • responsiveness to user needs

The steps in the Open Source model include:

  • Coding
  • Review
  • Pre-Commit Test
  • Development Release
  • Parallel Debugging
  • Production Release

Implementation Examples:

  • Incremental and decentralized integration in FreeBSD
  • The Apache HTTP Project

Features

Notable features of the Open Source model include:

  • a delegation of commit authority
  • users see the results early on in the development phase

Deficiencies

Notable 'deficiencies' of the Open Source model include:

  • working on source code that is public does not provide a revenue stream
  • the seed concept needs to have been developed to some extent before this model can be implemented


Design

Design is about the relationship of parts to the whole.  A program may work, but be poorly designed.  If the program models a real world situation, the program will more than likely be simpler to upgrade. 

Function-Oriented Design

Function-oriented gained popularity in North America:

  • hide the information
    • hide the representation of data
    • hide the properties of data
    • hide the solution structure
  • Structure Chart

  • avoid global variables - since one change in a global variable may affect many modules

Data-Oriented Design

Data-oriented gained popularity in Europe.  In the 1970's, Michael Jackson developed a modelling method based on structured programming principles: 

  • depict orderly motion of data from process to process
  • focus on data availability controlling the execution
  • Jackson Structure Diagrams - highlight the structure of the solution in terms of the three fundamental constructs: sequence, selection and iteration.

Decision tables organize a set of compound conditions into simple selection constructs.

 if ( balance - cheque - 0.50 >= 0 ) {
     balance -= cheque;
     balance -= 0.50;
 } else if ( !( balance - cheque - 0.50 >= 0 ) &&
             balance - cheque - 10.00 >= -1000.00 ) {
     balance -= cheque;
     balance -= 10.00;
 } else if ( !( balance - cheque - 0.50 >= 0 ) &&
             !( balance - cheque - 10.00 >= -1000.00 ) ) 
     balance -= 15.00;

Accounting for the exclusions in each test, this code can then be simplified to 

 if ( balance - cheque - 0.50 >= 0 ) {
     balance -= cheque;
     balance -= 0.50;
 } else if ( balance - cheque - 10.00 >= -1000.00 ) { 
     balance -= cheque;
     balance -= 10.00;
 } else
     balance -= 15.00;
